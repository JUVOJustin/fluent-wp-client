## Overview

`fluent-wp-client` supports the WordPress Abilities API exposed at `/wp-json/wp-abilities/v1`.

Use it to inspect registered abilities, list categories, execute abilities, and fetch the live input/output schemas exposed by WordPress.

## Metadata

```ts title="Ability metadata"
const abilities = await wp.abilities();
const ability = await wp.ability('test/get-site-title').definition();
const categories = await wp.abilityCategories();
const category = await wp.abilityCategory('test');
```

Use `ability(name)` for schema helpers too:

```ts title="Ability schema helpers"
const outputSchema = await wp.ability('test/get-site-title').getJsonSchema('output');
const inputValidator = await wp.ability('test/process-complex').getStandardSchema('input');
```

## Execution

Use `execute()` for every ability. The client reads the ability annotations and chooses the correct WordPress execution mode:

- `readonly: true` executes with `GET /run`
- `destructive: true` executes with `DELETE /run`
- other abilities execute with `POST /run`

```ts title="Execute abilities"
const siteTitle = await wp
  .ability<undefined, { title: string }>('test/get-site-title')
  .execute();

const updated = await wp
  .ability<{ value: string }, { previous: string; current: string }>('test/update-option')
  .execute({ value: 'hello' });

const deleted = await wp
  .ability<string, { deleted: boolean }>('test/delete-option')
  .execute('test_ability_option');
```

TypeScript cannot infer static input/output types from a runtime catalog by itself. For compile-time types, pass generics manually or use the CLI to generate TypeScript types from a known WordPress site. For runtime validation from the live catalog, use `getStandardSchema()`.

## Reusable Handles

Use `client.ability(name)` when you want a reusable ability handle.

```ts title="Fluent ability execution"
const processComplex = wp
  .ability<
    { name: string; settings: { theme: string } },
    { processed: boolean; echo: { name: string } }
  >('test/process-complex')
;

const result = await processComplex.execute({
  name: 'demo',
  settings: { theme: 'dark' },
});
```

Preferred builder methods:

- `definition()`
- `execute(input?)`
- `getJsonSchema(schema)`
- `getStandardSchema(schema)`

## Custom Validation

Ability execution does not run local validation. WordPress remains the source of truth.

When you want app-level validation, fetch the live schema and validate the payload yourself before calling the ability:

```ts title="Validate one ability input with the live schema"
import { z } from 'zod';

const inputJsonSchema = await wp.ability('test/process-complex').getJsonSchema('input');
const outputJsonSchema = await wp.ability('test/process-complex').getJsonSchema('output');
const inputSchema = z.fromJSONSchema(inputJsonSchema);
const outputSchema = z.fromJSONSchema(outputJsonSchema);

const validatedInput = inputSchema.parse({
  name: 'demo',
  settings: { theme: 'dark' },
});

const result = await wp.ability('test/process-complex').execute(validatedInput);
const validatedResult = outputSchema.parse(result);
```

## Schema Access

Prefer ability JSON Schemas from WordPress instead of static package-level ability validators:

```ts title="Ability JSON Schema options"
const definitionSchema = await wp.ability('test/get-site-title').getJsonSchema('definition');
const annotationsSchema = await wp.ability('test/get-site-title').getJsonSchema('annotations');
const inputSchema = await wp.ability('test/process-complex').getJsonSchema('input');
const outputSchema = await wp.ability('test/delete-option').getJsonSchema('output');
```

`input` and `output` come from the live ability description and apply regardless of whether the ability executes through GET, POST, or DELETE. `definition`, `annotations`, and `category` fall back to the built-in generic JSON Schemas when a live catalog schema is not available.

:::tip[Need native Zod APIs?]
Convert the discovered JSON Schema in application code when you need `.parse()`, `.shape`, or `.extend()`:

```ts
import { z } from 'zod';

const inputJsonSchema = await wp.ability('test/process-complex').getJsonSchema('input');
const inputSchema = z.fromJSONSchema(inputJsonSchema);
const input = inputSchema.parse({ name: 'demo', settings: { theme: 'dark' } });
```
:::

Fallback ability metadata schemas stay extensible so plugin-added fields stay available.

## Validation model

- Ability metadata helpers validate payloads with built-in ability schemas
- Ability execution defers request validation to WordPress
- Use `describe()` or `explore()` when you want local validation in your application
- Failed REST responses throw `WordPressHttpError` (subclass of `WordPressClientError`)

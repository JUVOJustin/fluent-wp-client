The `fluent-wp-client/ai-sdk` entrypoint provides tool factories that turn WordPress client methods into [Vercel AI SDK](https://sdk.vercel.ai/) tools. Each tool is imported and registered manually, giving you full control over which capabilities your AI agent has access to.

## Install

```bash title="Installing dependencies…"
npm install fluent-wp-client ai
```

`ai` is an optional peer dependency -- only required when importing from `fluent-wp-client/ai-sdk`.

## Quick start

```typescript title="tools.ts"
import { WordPressClient } from 'fluent-wp-client';
import { getPostsTool, getPostTool, createPostTool } from 'fluent-wp-client/ai-sdk';

const wp = new WordPressClient({
  baseUrl: 'https://example.com',
  auth: { username: 'admin', password: 'xxxx xxxx xxxx xxxx' },
});

const tools = {
  searchPosts: getPostsTool(wp, {
    description: 'Search published blog posts.',
    defaultArgs: { perPage: 5 },
    fixedArgs: { status: 'publish', fields: ['id', 'slug', 'title', 'excerpt', 'link'] },
  }),
  readPost: getPostTool(wp, {
    defaultArgs: { includeContent: true },
  }),
  draftPost: createPostTool(wp, {
    description: 'Create a new draft post for editorial review.',
    fixedInput: { status: 'draft' },
  }),
};
```

Pass these tools to `generateText`, `streamText`, or any AI SDK agent:

```typescript title="agent.ts"
import { generateText } from 'ai';
import { openai } from '@ai-sdk/openai';

const { text } = await generateText({
  model: openai('gpt-4o'),
  tools,
  prompt: 'Find the latest 3 posts about TypeScript.',
});
```

## Design principles

### Manual composition

There is no `.toAiSDK()` magic or auto-registration helper. You import individual factories, configure each one, and add it to your `tools` object. This is intentional:

- You control which operations the model can perform.
- Different tools can use different client instances (e.g. public vs. admin).
- Descriptions and defaults are tuned per tool, not globally.

### Lean tool surface

Each resource exposes just enough tools for a model to work effectively:

| Pattern | Tool | What the model can do |
|---|---|---|
| **Collection** | `getPostsTool` | Search, filter, paginate, sort, select fields |
| **Single item** | `getPostTool` | Fetch by ID or slug, optionally expand content/blocks |
| **Create** | `createPostTool` | Create a new resource |
| **Update** | `updatePostTool` | Update an existing resource |
| **Delete** | `deletePostTool` | Delete a resource |

A single `getPostsTool` replaces what would otherwise be separate search, list, and paginated tools. The model calls it with `{ search: "keyword" }`, `{ perPage: 5, page: 2 }`, or `{ categories: [3], orderby: "date" }` as needed.

### Auth inheritance

Every tool executes through the `WordPressClient` instance you pass to the factory. Auth headers, cookies, credentials, and custom `fetch` all carry through automatically. Tool input schemas intentionally exclude auth fields.

```typescript
const publicWp = new WordPressClient({ baseUrl });
const adminWp = new WordPressClient({ baseUrl, auth: { username, password } });

const tools = {
  searchPosts: getPostsTool(publicWp, { fixedArgs: { status: 'publish' } }),
  createPost: createPostTool(adminWp),
};
```

## Configuration options

Every tool factory accepts an options object with three configuration surfaces:

### `description`

Override the AI-facing tool description. Use this to give the model better context for when and how to call the tool.

```typescript
getPostsTool(wp, {
  description: 'Search the company knowledge base for internal articles.',
});
```

### `defaultArgs`

Provide fallback values the model can override. Useful for sensible defaults that shouldn't require explicit model input every time.

```typescript
getPostsTool(wp, {
  defaultArgs: { perPage: 5, status: 'publish' },
});
// Model calls with { search: "react" }
// Executes as { search: "react", perPage: 5, status: "publish" }
```

### `fixedArgs`

Lock values that always override model input. Use this to enforce constraints the model should not bypass.

```typescript
getPostsTool(wp, {
  fixedArgs: { status: 'publish', fields: ['id', 'slug', 'title', 'excerpt'] },
});
// Model calls with { status: "draft", perPage: 10 }
// Executes as { status: "publish", perPage: 10, fields: [...] }
```

### Mutation tools: `defaultInput` and `fixedInput`

Create and update tools wrap the write payload in an `input` object. `defaultInput` and `fixedInput` apply the same precedence logic to nested fields:

```typescript
createPostTool(adminWp, {
  description: 'Draft a new post for review.',
  fixedInput: { status: 'draft', author: 7 },
});
// Model provides: { input: { title: "New Article", content: "..." } }
// Executes as:    { title: "New Article", content: "...", status: "draft", author: 7 }
```

```typescript
updatePostTool(adminWp, {
  defaultInput: { status: 'draft' },
});
// Model provides: { id: 42, input: { title: "Edited" } }
// Executes as:    id=42, { title: "Edited", status: "draft" }
// Model provides: { id: 42, input: { title: "Final", status: "publish" } }
// Executes as:    id=42, { title: "Final", status: "publish" }  (model wins)
```

## Instance-aware Zod schemas for AI tools

The built-in tool factories use generic input schemas that cover core WordPress fields. For tighter validation that reflects your actual WordPress instance — including custom post types, ACF fields, custom taxonomies, and plugin-registered query parameters — convert the discovered JSON Schema to Zod.

<Tabs>
<TabItem label="Runtime (dynamic)">

Use runtime conversion when the WordPress instance is not known at build time — multi-tenant apps, user-provided sites, or agents that connect to different instances.

```ts title="Discover and validate at runtime"
import { generateObject } from 'ai';
import { openai } from '@ai-sdk/openai';
import { WordPressClient } from 'fluent-wp-client';
import { zodSchemasFromDescription } from 'fluent-wp-client/zod';

const wp = new WordPressClient({
  baseUrl: siteUrl,
  auth: { username, password },
});

// Discover the actual schema for this instance
const catalog = await wp.explore({ include: ['content'] });
const postSchemas = zodSchemasFromDescription(catalog.content.posts);

// The model gets the real create schema — including custom fields
const { object } = await generateObject({
  model: openai('gpt-4o-mini'),
  schema: postSchemas.create!,
  prompt: 'Draft a short release announcement post.',
});

await wp.content('posts').create(object);
```

For abilities:

```ts title="Discover ability input schema at runtime"
import { zodSchemasFromDescription } from 'fluent-wp-client/zod';

const catalog = await wp.explore({ include: ['abilities'] });
const calcSchemas = zodSchemasFromDescription(catalog.abilities['acme/calculate']);

// Use the discovered input schema as a tool's inputSchema
const calculateTool = tool({
  description: 'Run a calculation via the WordPress ability API.',
  inputSchema: calcSchemas.input!,
  execute: async (input) => wp.ability('acme/calculate').run(input),
});
```
</TabItem>
<TabItem label="CLI (static)">

Use CLI-generated schemas when your app connects to a single known WordPress instance. The generated file gives you compile-time type safety and tree-shakeable validators.

```bash title="Generate schema files"
npx fluent-wp-client schemas \
  --url https://example.com \
  --username admin \
  --password "xxxx xxxx xxxx xxxx" \
  --format zod \
  --zod-out src/lib/wp-schemas.ts
```

```ts title="Use generated schemas with AI SDK"
import { generateObject } from 'ai';
import { openai } from '@ai-sdk/openai';
import { wpPostSchema } from './lib/wp-schemas';

const { object } = await generateObject({
  model: openai('gpt-4o-mini'),
  schema: wpPostSchema,
  prompt: 'Draft a short release announcement post.',
});

await adminWp.content('posts').create(object);
```

:::note
Regenerate the schema file when your WordPress instance changes (new custom fields, plugins, etc.):
```bash
npx fluent-wp-client schemas --url https://example.com --format zod --zod-out src/lib/wp-schemas.ts
```
:::
</TabItem>
</Tabs>

### When to use which

| Scenario | Approach |
|----------|----------|
| Multi-tenant platform, user-provided WordPress URLs | **Runtime** — `explore()` + `zodSchemasFromDescription()` |
| AI agent that adapts to the connected site | **Runtime** — schema reflects actual capabilities |
| Single known WordPress instance | **CLI** — compile-time types, no runtime discovery |
| CI validation or shared schema across services | **CLI** — checked-in schema file |

## Single item getters

Post-like getter tools (`getPostTool`, `getPageTool`) return a content envelope instead of the raw DTO:

```typescript
const tool = getPostTool(wp, { defaultArgs: { includeContent: true } });

// Model calls: { id: 42, includeContent: true, includeBlocks: true }
// Returns:
// {
//   item: { id: 42, title: { rendered: "..." }, ... },
//   content: { raw: "<!-- wp:paragraph -->...", rendered: "<p>...", protected: false },
//   blocks: [{ blockName: "core/paragraph", ... }]
// }
```

Non-content resources (`getCategoryTool`, `getTagTool`, etc.) return the DTO directly.

All single-item tools accept either `id` or `slug`:

```typescript
// By ID
{ id: 42 }

// By slug
{ slug: 'hello-world' }
```

## Field selection

Collection tools and single-item tools support `fields` to limit the response payload. This reduces token usage in agent conversations:

```typescript
getPostsTool(wp, {
  fixedArgs: { fields: ['id', 'slug', 'title', 'excerpt', 'link'] },
});
```

The model can also request field selection at runtime. Both `fields` and `_fields` are accepted and normalized automatically.

## Full example

```typescript title="wordpress-agent.ts"
import { generateText } from 'ai';
import { openai } from '@ai-sdk/openai';
import { WordPressClient } from 'fluent-wp-client';
import {
  getPostsTool,
  getPostTool,
  getCategoriesTool,
  createPostTool,
  updatePostTool,
} from 'fluent-wp-client/ai-sdk';

const publicWp = new WordPressClient({
  baseUrl: 'https://blog.example.com',
});

const adminWp = new WordPressClient({
  baseUrl: 'https://blog.example.com',
  auth: { username: 'editor', password: 'xxxx xxxx xxxx xxxx' },
});

const tools = {
  searchPosts: getPostsTool(publicWp, {
    description: 'Search published blog posts by keyword, category, or tag.',
    defaultArgs: { perPage: 5 },
    fixedArgs: { status: 'publish', fields: ['id', 'slug', 'title', 'excerpt', 'link', 'categories'] },
  }),
  readPost: getPostTool(publicWp, {
    description: 'Read the full content of a blog post by ID or slug.',
    defaultArgs: { includeContent: true },
  }),
  listCategories: getCategoriesTool(publicWp, {
    description: 'List all blog categories.',
  }),
  draftPost: createPostTool(adminWp, {
    description: 'Create a new draft blog post for editorial review.',
    fixedInput: { status: 'draft' },
  }),
  publishPost: updatePostTool(adminWp, {
    description: 'Publish an existing draft post.',
    fixedInput: { status: 'publish' },
  }),
};

const { text } = await generateText({
  model: openai('gpt-4o'),
  tools,
  maxSteps: 10,
  prompt: 'Find all posts about TypeScript in the Technology category, then draft a summary post.',
});
```

## Overview

`fluent-wp-client` supports static auth and request-aware auth.

Use this guide to choose the right approach for your runtime, security model, and request lifecycle.

## Recommended default

For most applications, do not start with resolver helpers.

Resolve auth in your app integration layer (middleware, request context, service container), create a `WordPressClient` with that final auth, and pass the client through your handlers.

This is usually easier to reason about, easier to test, and easier to debug than adding an extra resolver abstraction.

```ts title="Default pattern: resolve auth in middleware, then create client"
export function createWordPressClientForRequest(request: Request): WordPressClient {
  const token = readUserJwtFromRequest(request);

  return new WordPressClient({
    baseUrl: process.env.WP_BASE_URL!,
    auth: token ? { token } : undefined,
  });
}
```

## Which method should I use?

| Use case | Best option | Why |
| --- | --- | --- |
| Public read-only content | No auth | No credentials required for public endpoints |
| Backend jobs, admin scripts, server-side tools | Basic auth | Stable server-to-server access with WordPress application passwords |
| SPA/mobile or external token flow | JWT auth | Stateless bearer token model |
| Logged-in WordPress browser session | Cookie + nonce auth | Matches WordPress native browser auth (`X-WP-Nonce` + cookies) |
| You already have a final `Authorization` value | `authHeader` or `auth.authorization` | Reuse existing header without re-building credentials |
| Signature/custom HMAC/OAuth-like auth | `authHeaders` provider | Compute dynamic headers per request |
| Request-scoped auth selection in your own wrapper | `RequestAuthResolver` and `createAuthResolver` | Keep resolver functions typed and reusable |

## Request-aware auth headers

Use `authHeaders` when auth data must be computed for each request.

```ts title="Request signing"
const wp = new WordPressClient({
  baseUrl: 'https://example.com',
  authHeaders: ({ method, url, body }) => ({
    Authorization: createSignedAuthHeader({ method, url: url.toString(), body }),
  }),
});
```

This is the right choice for timestamped signatures, HMAC headers, rotating API keys, and similar schemes.

## Resolver helpers

`RequestAuthResolver` and `createAuthResolver` are selector helpers for your framework layer.

They do not perform authentication by themselves.
They only return one auth shape (`{ token }`, `{ username, password }`, `{ nonce, credentials }`, header auth, or `null`) based on request context.

Use this mental model:

1. Your resolver decides which auth method applies for this request.
2. You pass the resolved auth value into client calls.
3. The client applies headers/credentials for that concrete auth value.

- Use `RequestAuthResolver` when you want a named, reusable resolver function type.
- Use `createAuthResolver` when you want an identity helper that preserves generic type inference for resolver context.

```ts title="Typed resolver helper"
type RequestContext = {
  token?: string;
  nonce?: string;
  isBrowser: boolean;
};

const resolveRequestAuth = createAuthResolver<RequestContext>((context) => {
  if (context.isBrowser && context.nonce) {
    return { nonce: context.nonce, credentials: 'include' };
  }

  if (context.token) {
    return { token: context.token };
  }

  return null;
});
```

```ts title="Use resolver output per request"
const resolvedAuth = await resolveRequestAuth(context);

const post = await wp.request({
  endpoint: '/posts/1',
  auth: resolvedAuth ?? undefined,
});
```

Choose these helpers when auth depends on who is making the request, where the request runs, or how your app stores credentials.

## Real-world examples

### SSR middleware: Resolve auth from request context

`ResolvableWordPressAuth<TContext>` lets you keep the client itself framework-agnostic while resolving auth from the server context that your app already has.

Use `RequestAuthResolver` when your handler receives a web-standard `Request`, or `createAuthResolver()` when you want typed inference for a richer framework context.

<Tabs>
  <TabItem label="Generic">

```ts title="Generic Request handlers"
import {
  WordPressClient,
  createAuthResolver,
  resolveWordPressAuth,
  type RequestAuthResolver,
} from 'fluent-wp-client';

function readCookie(cookieHeader: string | null, name: string): string | null {
  if (!cookieHeader) {
    return null;
  }

  const cookie = cookieHeader
    .split(';')
    .map((segment) => segment.trim())
    .find((segment) => segment.startsWith(`${name}=`));

  return cookie ? decodeURIComponent(cookie.slice(name.length + 1)) : null;
}

const requestAuthResolver: RequestAuthResolver = createAuthResolver((request) => {
  const jwt = readCookie(request.headers.get('cookie'), 'wp_jwt');

  if (jwt) {
    return { token: jwt };
  }

  const sessionId = request.headers.get('x-session-id');

  if (sessionId) {
    return { authorization: `Session ${sessionId}` };
  }

  if (process.env.WP_APP_PASSWORD) {
    return {
      username: process.env.WP_APP_USER ?? 'admin',
      password: process.env.WP_APP_PASSWORD,
    };
  }

  return null;
});

export async function handleRequest(request: Request) {
  const auth = await resolveWordPressAuth(requestAuthResolver, request);

  const wp = new WordPressClient({
    baseUrl: 'https://your-wordpress-site.com',
    auth,
  });

  return wp.getCurrentUser();
}
```

This pattern covers common SSR fallbacks:

- Cookie-to-JWT token extraction
- Session ID forwarding from request headers
- Application passwords from server environment variables

  <TabItem label="Astro">

```ts title="Astro Actions"
import type { ActionAPIContext } from 'astro:actions';
import {
  createAuthResolver,
  resolveWordPressAuth,
  type ResolvableWordPressAuth,
} from 'fluent-wp-client';

export type ActionAuthConfig = ResolvableWordPressAuth<ActionAPIContext>;

export const actionAuth = createAuthResolver<ActionAPIContext>((context) => {
  const jwt = context.cookies.get('wp_jwt')?.value;

  if (jwt) {
    return { token: jwt };
  }

  const sessionId = context.request.headers.get('x-session-id');

  if (sessionId) {
    return { authorization: `Session ${sessionId}` };
  }

  if (process.env.WP_APP_PASSWORD) {
    return {
      username: process.env.WP_APP_USER ?? 'admin',
      password: process.env.WP_APP_PASSWORD,
    };
  }

  return null;
});

export async function resolveActionRequestAuth(
  config: { auth?: ActionAuthConfig },
  context: ActionAPIContext,
) {
  return config.auth ? await resolveWordPressAuth(config.auth, context) : null;
}
```

This mirrors the pattern used by Astro integrations: keep `auth` configurable, then resolve it against the current `ActionAPIContext` right before you build request headers.

```ts title="Next.js App Router"
import {
  WordPressClient,
  createAuthResolver,
  resolveWordPressAuth,
} from 'fluent-wp-client';

type NextRouteContext = {
  request: Request;
  params: { slug: string };
};

const nextRouteAuth = createAuthResolver<NextRouteContext>(({ request }) => {
  const bearer = request.headers.get('authorization');

  if (bearer) {
    return bearer;
  }

  const jwtCookie = request.headers
    .get('cookie')
    ?.split(';')
    .map((segment) => segment.trim())
    .find((segment) => segment.startsWith('wp_jwt='))
    ?.slice('wp_jwt='.length);

  return jwtCookie ? { token: decodeURIComponent(jwtCookie) } : null;
});

export async function GET(request: Request, { params }: { params: { slug: string } }) {
  const auth = await resolveWordPressAuth(nextRouteAuth, { request, params });

  const wp = new WordPressClient({
    baseUrl: process.env.WORDPRESS_BASE_URL!,
    auth,
  });

  return Response.json(await wp.content('posts').getBySlug(params.slug));
}
```

### Example A: Typical app backend (recommended, no resolver)

Use middleware auth once and create one request-scoped client.

```ts title="Express-style request-scoped client"
app.use(async (req, _res, next) => {
  const token = req.user?.wpJwt;

  req.wp = new WordPressClient({
    baseUrl: process.env.WP_BASE_URL!,
    auth: token ? { token } : undefined,
  });

  next();
});
```

### Example B: Multi-runtime app gateway (resolver helps)

Use resolver helpers when one shared integration must select auth per request based on runtime and actor.

```ts title="Shared auth selector for browser, user token, or service auth"
type GatewayContext = {
  isBrowser: boolean;
  wpNonce?: string;
  userJwt?: string;
  isInternalServiceCall: boolean;
};

const resolveAuth: RequestAuthResolver<GatewayContext> = createAuthResolver((context) => {
  if (context.isBrowser && context.wpNonce) {
    return { nonce: context.wpNonce, credentials: 'include' };
  }

  if (context.userJwt) {
    return { token: context.userJwt };
  }

  if (context.isInternalServiceCall) {
    return {
      username: process.env.WP_APP_USER!,
      password: process.env.WP_APP_PASSWORD!,
    };
  }

  return null;
});
```

### Example C: Signed plugin endpoint (use `authHeaders`, not resolver)

Use `authHeaders` when every request needs computed signature headers.

```ts title="HMAC/timestamp signature per request"
const wp = new WordPressClient({
  baseUrl: 'https://example.com',
  authHeaders: ({ method, url, body }) => {
    const timestamp = new Date().toISOString();

    return {
      Authorization: createSignedAuthHeader({ method, url: url.toString(), body, timestamp }),
      'X-Signature-Timestamp': timestamp,
    };
  },
});
```

## Header utility helpers

Use the exported helpers when you need to build auth headers outside client construction:

- `createBasicAuthHeader(credentials)`
- `createJwtAuthHeader(credentialsOrToken)`
- `createWordPressAuthHeader(authInput)`

## JWT login lifecycle helpers

Use these when your app obtains and verifies JWT tokens:

- `loginWithJwt({ username, password })`
- `validateJwtToken(token?)`

## Overview

When using relationship hydration methods like the fluent `content(resource).item(...).with(...)` API, the client automatically fetches embedded data and transforms it into typed, easy-to-use relations. Unlike raw `embed: true` which exposes WordPress's internal `_embedded` structure, `.with()` provides a clean, type-safe API with automatic fallbacks.

## `embed: true` vs `.with()`: What's the difference?

### Raw embedded data (`embed: true`)

WordPress REST API can include embedded entities when you add `?_embed=true`. This returns raw `_embedded` data that mirrors WordPress's internal link structure:

```typescript
// Request with embed: true
const posts = await client.content('posts').list({ perPage: 10, embed: true });

// Access raw embedded data (WordPress internal structure)
posts[0]._embedded.author[0]              // author object
posts[0]._embedded['wp:term']             // mixed categories + tags array
posts[0]._embedded['wp:featuredmedia'][0] // featured media object
posts[0]._embedded['acf:post']            // ACF relationship data
```

**Characteristics of raw embed:**
- WordPress internal naming (`wp:term`, `wp:featuredmedia`, `acf:post`)
- No type safety—data is `any` type
- No fallback if `_embedded` is missing
- Requires manual parsing of taxonomy arrays
- Plugin data mixed in shared buckets

### Client-side relation hydration (`.with()`)

The `.with()` method provides intelligent hydration that transforms raw embedded data (or falls back to API calls) into clean, typed relations:

```typescript
// Request with .with()
const post = await client.content('posts').item('my-post').with('author', 'categories', 'tags');

// Access typed relations
post.related.author        // WordPressAuthor | null (fully typed)
post.related.categories    // WordPressCategory[] (clean array)
post.related.tags          // WordPressTag[] (separate from categories)
```

**Characteristics of `.with()`:**
- Clean, semantic naming (`author`, `categories`, `featuredMedia`)
- Full TypeScript type safety
- Automatic fallback to API calls if `_embedded` missing
- Separates taxonomies into clean arrays
- Supports custom relations via registry

## Comparison Table

| Feature | `embed: true` | `.with()` |
|---------|--------------|-----------|
| **Type safety** | ❌ Manual casting | ✅ Fully typed |
| **Property names** | WordPress internal (`wp:term`, `wp:featuredmedia`) | Clean names (`categories`, `featuredMedia`) |
| **Taxonomy handling** | Mixed in `_embedded['wp:term']` | Separated `categories` and `tags` arrays |
| **Fallback support** | ❌ No fallback | ✅ Automatic API calls using ID fields |
| **Custom relations** | Manual parsing | ✅ Register once, use everywhere |
| **ACF support** | Parse `_embedded['acf:post']` manually | ✅ `registerAcfRelationship()` helpers |
| **Null safety** | Check array lengths manually | ✅ Graceful null/empty arrays |

## Why Fields Are Automatically Preserved

Relationship hydration works by extracting embedded data from the `_embedded` property of REST API responses. When embedded data is unavailable, the client falls back to making separate API calls using ID fields (`author`, `categories`, `tags`, `featured_media`).

If these fields were excluded from the response, relationship hydration would fail. Therefore, the client automatically:

1. **Requests `_embed`** from the API when `.with()` is called (even though it's stripped from final output by default)
2. **Preserves ID fields** required for fallback fetching
3. **Hydrates `related` property** with typed, clean data

## Critical Fields for Each Relation

| Relation | Required ID Field | Embedded Data Path | Result Type |
|----------|-------------------|-------------------|-------------|
| `author` | `author` | `_embedded.author` | `WordPressAuthor \| null` |
| `categories` | `categories` (array) | `_embedded['wp:term']` (filtered) | `WordPressCategory[]` |
| `tags` | `tags` (array) | `_embedded['wp:term']` (filtered) | `WordPressTag[]` |
| `terms` | `categories` + `tags` | Same as above | `{ categories, tags, taxonomies }` |
| `featuredMedia` | `featured_media` | `_embedded['wp:featuredmedia']` | `WordPressMedia \| null` |
| `parent` | `parent` | `_embedded['up']` | `WordPressPostLike \| null` |

## How It Works

### Automatic `_embed` Handling

When you call `.with()`, the client automatically requests `_embed=true` from the API for hydration, but strips it from the final response to keep payloads lean (unless you explicitly pass `embed: true`):

```typescript
// _embed is requested for hydration, but stripped from output
const post = await client.content('posts').item('my-post').with('author', 'categories');

// post does NOT have _embedded property
// post.related.author and post.related.categories ARE populated
```

To keep `_embedded` in the response, explicitly request it:

```typescript
// Both _embedded AND related are present
const post = await client.content('posts').item('my-post', { embed: true }).with('author');
```

### Field Preservation

Even if you use field filtering, the client ensures required fields are present for hydration:

```typescript
// The fields filter restricts the response
const posts = await client.content('posts').list({ fields: ['id', 'title'] });

// But when using relations, required fields are preserved
const post = await client.content('posts').item('my-post').with('author');
// author field is preserved even if not in the original fields list
```

### Fallback Mechanism

If `_embedded` data is missing but ID fields are present, the client makes separate API calls:

```typescript
const post = await client.content('posts').item('my-post').with('author');

// If _embedded.author is missing:
// 1. Uses post.author ID to fetch author separately
// 2. Resolves to the full author object
// 3. Returns it in post.related.author
```

This is a key advantage over raw `embed: true`—if WordPress doesn't include embedded data (auth restrictions, plugins, etc.), `.with()` still works via fallback API calls.

## Edge Cases

### Missing ID Fields

If the WordPress REST API masks certain fields (e.g., author IDs for unauthenticated users), the client gracefully handles this:

```typescript
// Public client may receive author: 0
const post = await publicClient.content('posts').item('my-post').with('author');
// post.related.author will be null (not an error)
```

### Empty Relations

Posts without certain relations are handled gracefully:

```typescript
// Post without categories
const post = await client.content('posts').item('my-post').with('categories');
// post.related.categories will be an empty array []

// Post without featured media
const post = await client.content('posts').item('my-post').with('featuredMedia');
// post.related.featuredMedia will be null
```

### Multiple Relations

When requesting multiple relations, the client efficiently batches fallback requests:

```typescript
const post = await client.content('posts')
  .item('my-post')
  .with('author', 'categories', 'tags', 'featuredMedia');

// Embedded data is used where available
// Fallback API calls are batched for missing embedded data
// All resolved in parallel for optimal performance
```

## Best Practices

1. **Prefer `.with()` over `embed: true`** for most use cases—it provides type safety, cleaner code, and automatic fallbacks
2. **Use `embed: true`** only when you need access to plugin-specific embedded data that doesn't fit standard relations
3. **Combine both** when you need raw plugin data AND typed standard relations: `.item('slug', { embed: true }).with('author')`
4. **Let the client handle fields**: Don't manually exclude critical fields when using relations—the client preserves what's needed
5. **Check for null**: Always handle cases where relations may be null (e.g., no featured media)
6. **Auth for full data**: Use authenticated clients to get complete author information

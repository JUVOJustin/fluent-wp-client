### Resolve auth from SSR request context

`ResolvableWordPressAuth<TContext>` lets you keep the client itself framework-agnostic while resolving auth from the server context that your app already has.

Use `RequestAuthResolver` when your handler receives a web-standard `Request`, or `createAuthResolver()` when you want typed inference for a richer framework context.

<Tabs>
  <TabItem label="Generic">

```ts title="Generic Request handlers"
import {
  WordPressClient,
  createAuthResolver,
  resolveWordPressAuth,
  type RequestAuthResolver,
} from 'fluent-wp-client';

function readCookie(cookieHeader: string | null, name: string): string | null {
  if (!cookieHeader) {
    return null;
  }

  const cookie = cookieHeader
    .split(';')
    .map((segment) => segment.trim())
    .find((segment) => segment.startsWith(`${name}=`));

  return cookie ? decodeURIComponent(cookie.slice(name.length + 1)) : null;
}

const requestAuthResolver: RequestAuthResolver = createAuthResolver((request) => {
  const jwt = readCookie(request.headers.get('cookie'), 'wp_jwt');

  if (jwt) {
    return { token: jwt };
  }

  const sessionId = request.headers.get('x-session-id');

  if (sessionId) {
    return { authorization: `Session ${sessionId}` };
  }

  if (process.env.WP_APP_PASSWORD) {
    return {
      username: process.env.WP_APP_USER ?? 'admin',
      password: process.env.WP_APP_PASSWORD,
    };
  }

  return null;
});

export async function handleRequest(request: Request) {
  const auth = await resolveWordPressAuth(requestAuthResolver, request);

  const wp = new WordPressClient({
    baseUrl: 'https://your-wordpress-site.com',
    auth,
  });

  return wp.getCurrentUser();
}
```

This pattern covers common SSR fallbacks:

- cookie-to-JWT token extraction
- session ID forwarding from request headers
  - application passwords from server environment variables

```ts title="Astro Actions"
import type { ActionAPIContext } from 'astro:actions';
import {
  createAuthResolver,
  resolveWordPressAuth,
  type ResolvableWordPressAuth,
} from 'fluent-wp-client';

export type ActionAuthConfig = ResolvableWordPressAuth<ActionAPIContext>;

export const actionAuth = createAuthResolver<ActionAPIContext>((context) => {
  const jwt = context.cookies.get('wp_jwt')?.value;

  if (jwt) {
    return { token: jwt };
  }

  const sessionId = context.request.headers.get('x-session-id');

  if (sessionId) {
    return { authorization: `Session ${sessionId}` };
  }

  if (process.env.WP_APP_PASSWORD) {
    return {
      username: process.env.WP_APP_USER ?? 'admin',
      password: process.env.WP_APP_PASSWORD,
    };
  }

  return null;
});

export async function resolveActionRequestAuth(
  config: { auth?: ActionAuthConfig },
  context: ActionAPIContext,
) {
  return config.auth ? await resolveWordPressAuth(config.auth, context) : null;
}
```

This mirrors the pattern used by Astro integrations: keep `auth` configurable, then resolve it against the current `ActionAPIContext` right before you build request headers.

```ts title="Next.js App Router"
import {
  WordPressClient,
  createAuthResolver,
  resolveWordPressAuth,
} from 'fluent-wp-client';

type NextRouteContext = {
  request: Request;
  params: { slug: string };
};

const nextRouteAuth = createAuthResolver<NextRouteContext>(({ request }) => {
  const bearer = request.headers.get('authorization');

  if (bearer) {
    return bearer;
  }

  const jwtCookie = request.headers
    .get('cookie')
    ?.split(';')
    .map((segment) => segment.trim())
    .find((segment) => segment.startsWith('wp_jwt='))
    ?.slice('wp_jwt='.length);

  return jwtCookie ? { token: decodeURIComponent(jwtCookie) } : null;
});

export async function GET(request: Request, { params }: { params: { slug: string } }) {
  const auth = await resolveWordPressAuth(nextRouteAuth, { request, params });

  const wp = new WordPressClient({
    baseUrl: process.env.WORDPRESS_BASE_URL!,
    auth,
  });

  return Response.json(await wp.getPostBySlug(params.slug));
}
```

## Method reference

### Read helpers

- `getPosts`, `getAllPosts`, `getPostsPaginated`, `getPost`, `getPostBySlug`
- `getPages`, `getAllPages`, `getPagesPaginated`, `getPage`, `getPageBySlug`
- `getMedia`, `getAllMedia`, `getMediaPaginated`, `getMediaItem`, `getMediaBySlug`, `getImageUrl`
- `getCategories`, `getAllCategories`, `getCategoriesPaginated`, `getCategory`, `getCategoryBySlug`
- `getTags`, `getAllTags`, `getTagsPaginated`, `getTag`, `getTagBySlug`
- `getUsers`, `getAllUsers`, `getUsersPaginated`, `getUser`, `getCurrentUser`
- `getComments`, `getAllComments`, `getCommentsPaginated`, `getComment`
- `getSettings`
- `searchContent(query, filter?)`

`getPost`, `getPostBySlug`, `getPage`, and `getPageBySlug` return promise-like queries.
You can still `await` them directly, and you can also call:

- `.get()` for explicit resource resolution
- `.getContent()` for `{ raw, rendered, protected }` via `context=edit`
- `.getBlocks()` for parsed Gutenberg blocks

Post/page list methods (`getPosts`, `getAllPosts`, `getPages`, etc.) return plain serializable DTO arrays without block helpers. Use a single-item query for block access:

```ts
const posts = await wp.getPosts({ perPage: 5 });
const blocks = await wp.getPost(posts[0].id).getBlocks();
```

Read helpers also accept request-scoped custom headers as the last argument:

```ts
const posts = await wp.getPosts(
  { perPage: 5 },
  {
    headers: {
      'x-crossposter-source-entity-type': 'post',
    },
  },
);
```

Media uses a different schema (`description`/`caption`, no `content.raw`), so media helpers do not expose `getBlocks()`.

### Generic resource helpers

- Content collections: `getContentCollection`, `getAllContentCollection`, `getContentCollectionPaginated`, `getContent`, `getContentBySlug`
- Term collections: `getTermCollection`, `getAllTermCollection`, `getTermCollectionPaginated`, `getTerm`, `getTermBySlug`
- Builders: `content(resource, schema?)`, `terms(resource, schema?)`

`content(resource)` and `terms(resource)` also expose unified pagination helpers:

- `.list(filter?)`
- `.listAll(filter?)`
- `.listPaginated(filter?)`

If `page`/`perPage` are omitted, endpoint defaults are preserved.

Passing a schema to `content(resource, schema)` validates generic content reads and mutations for that resource client.

All collection helpers (`getContentCollection`, `getAllContentCollection`, the `content()` builder, and built-in helpers like `getPosts`) accept a `search` string to filter results by keyword:

```ts
// Built-in helpers
const posts = await wp.getPosts({ search: 'hello world' });
const allPosts = await wp.getAllPosts({ search: 'hello world' });

// Generic collection helpers
const books = await wp.getContentCollection('books', { search: 'Test Book' });
const allBooks = await wp.getAllContentCollection('books', { search: 'Test Book' });

// content() builder
const bookResults = await wp.content('books').list({ search: 'Test Book' });
const allBookResults = await wp.content('books').listAll({ search: 'Test Book' });
```

### Extensible collection filters

Built-in collection helpers keep their typed core filters and also accept extra endpoint-specific params for plugin routes or project-specific REST extensions.

Unknown keys are forwarded through the normal query serializer, so camelCase filter names become snake_case query params and array values are emitted as repeated `param[]` entries.

```ts
const posts = await wp.getPosts({
  search: 'hello world',
  include: [4, 8],
  slug: ['hello-world', 'release-notes'],
  titleSearch: 'Hello',
});

const books = await wp.content('books').list({
  include: [164, 165],
  titleSearch: 'Test Book',
});
```

Use the typed helper when the endpoint shape is known, and drop to `route().param()` or `request()` when a plugin expects exact raw param names that should not be camelCase-normalized.

### Post-like typing

Use `WordPressCustomPost<TExtra>` or `WordPressPostLike` for CPTs where WordPress supports may disable `title`, `content`, `excerpt`, or `author`. Use `WordPressPostBase` only when the post type is guaranteed to expose content-bearing fields.

```ts
import type { WordPressCustomPost, WordPressPostBase } from 'fluent-wp-client';

// Sparse CPT — title, content, and excerpt are optional.
type Artifact = WordPressCustomPost<{ type: 'artifact' }>;

// Content-bearing CPT — title, content, and excerpt are required.
type ContentBook = WordPressPostBase & { type: 'book' };
```

### Mutation helpers

- Content: `createContent`, `updateContent`, `deleteContent`
- Terms: `createTerm`, `updateTerm`, `deleteTerm`
- Posts/pages: `createPost`, `updatePost`, `deletePost`, `createPage`, `updatePage`, `deletePage`
- Taxonomy built-ins: `createCategory`, `updateCategory`, `deleteCategory`, `createTag`, `updateTag`, `deleteTag`
- Users: `createUser`, `updateUser`, `deleteUser`
- Comments: `createComment`, `updateComment`, `deleteComment`
- Media: `createMedia`, `uploadMedia`, `updateMedia`, `deleteMedia`
- Settings: `updateSettings`

Helpers accept request-scoped custom headers so you can pass one-off metadata without mutating shared client defaults.

```ts
const createdPost = await wp.createPost(
  { title: 'Post title', status: 'draft' },
  {
    headers: {
      'x-crossposter-source-base-url': 'https://source.example',
      'x-crossposter-source-entity-type': 'post',
      'x-crossposter-source-entity-id': '42',
    },
  },
);

await wp.deletePost(createdPost.id, {
  force: true,
  headers: {
    'x-crossposter-source-entity-id': '42',
  },
});
```

`create*` / `update*` helper signatures support both patterns:

- `createPost(input, responseSchema?)`
- `createPost(input, requestOptions?)`
- `createPost(input, responseSchema?, requestOptions?)`

Auth overrides are intentionally blocked on mutation helper options. If you need different auth credentials or auth headers, create a new `WordPressClient` instance with the desired auth configuration.

The same auth-override rule applies to read and listing helpers.

### Auth and ability helpers

- `loginWithJwt(credentials)`
- `validateJwtToken(token?)`
- `getAbilities()` / `getAbility(name)`
- `getAbilityCategories()` / `getAbilityCategory(slug)`
- `executeGetAbility(name, input?, responseSchema?)`
- `executeRunAbility(name, input?, responseSchema?)`
- `executeDeleteAbility(name, input?, responseSchema?)`
- `ability<TInput, TOutput>(name)`

### Relation helpers

- `post(idOrSlug).with(...relations).get()`
- `getPostWithRelations(idOrSlug, ...relations)`
- `content(resource).item(idOrSlug).with(...relations).get()`
- `content(resource).getWithRelations(idOrSlug, ...relations)`

Supported relation names:

- `author`
- `categories`
- `tags`
- `terms`
- `featuredMedia`

`terms` always returns core `categories` and `tags` arrays when available, plus `terms.taxonomies` for generic taxonomy access. For custom post types, custom taxonomies such as `genre` are exposed under `terms.taxonomies.<taxonomy>`.

### Transport and request control

- `request(options)`
- `fetchAPI(endpoint, params?)`
- `fetchAPIPaginated(endpoint, params?)`
- `setHeaders(name, value)` / `setHeaders(headers)`
- `hasAuth()`

## CRUD walkthrough

### Posts

```ts
const createdPost = await wp.createPost({ title: 'Post title', status: 'draft' });

const updatedPost = await wp.updatePost(createdPost.id, {
  title: 'Post title updated',
  status: 'publish',
});

await wp.deletePost(createdPost.id, { force: true });
```

### Gutenberg block parsing

```ts
const postQuery = wp.getPostBySlug('hello-world');
const pageQuery = wp.getPageBySlug('about');

const postBlocks = await postQuery.getBlocks();
const pageBlocks = await pageQuery.getBlocks();

const postList = await wp.getPosts({ perPage: 5 });
const firstListItemBlocks = await wp.getPost(postList[0].id).getBlocks();
```

`getBlocks()` requires edit capabilities because it reads `content.raw` using `context=edit`.
When credentials cannot access edit context, WordPress returns the expected auth/capability error.

### Pages

```ts
const createdPage = await wp.createPage({ title: 'About us', status: 'draft' });

await wp.updatePage(createdPage.id, { status: 'publish' });
await wp.deletePage(createdPage.id, { force: true });
```

### Categories and tags

```ts
const category = await wp.createCategory({ name: 'News' });
const tag = await wp.createTag({ name: 'featured' });

await wp.updateCategory(category.id, { name: 'Company News' });
await wp.updateTag(tag.id, { name: 'featured-updated' });

await wp.deleteCategory(category.id, { force: true });
await wp.deleteTag(tag.id, { force: true });
```

### Comments

```ts
const post = await wp.getPostBySlug('hello-world');

if (!post) {
  throw new Error('Post not found.');
}

const comment = await wp.createComment({
  post: post.id,
  content: 'Nice post!',
  status: 'approve',
});

await wp.updateComment(comment.id, { content: 'Nice post, updated.' });
await wp.deleteComment(comment.id, { force: true });
```

### Media upload

```ts
const media = await wp.uploadMedia({
  file: imageBlob,
  filename: 'cover.jpg',
  mimeType: 'image/jpeg',
  title: 'Cover image',
  alt_text: 'Book cover image',
}, {
  headers: {
    'x-crossposter-source-entity-id': 'book-42',
  },

await wp.updateMedia(media.id, { caption: 'Updated caption' });
await wp.deleteMedia(media.id, { force: true });
```

### Generic CPT CRUD

```ts

  force: true,
  headers: {
```

### WPAPI-style CRUD chain


## Cross-resource search

`searchContent(query, filter?)` searches across posts, pages, and custom post types using the WordPress `/wp/v2/search` endpoint.

```ts
import type { WordPressSearchResult } from 'fluent-wp-client';

// Basic keyword search
const results = await wp.searchContent('hello');

// Filter by resource type
const posts = await wp.searchContent('hello', { type: 'post' });

// Filter by a single subtype (post type)
const pages = await wp.searchContent('about', {
  type: 'post',
  subtype: 'page',
});

// Filter by multiple subtypes — serialised as subtype[]=post&subtype[]=page&subtype[]=book
const mixed = await wp.searchContent('test', {
  type: 'post',
  subtype: ['post', 'page', 'book'],
});

// Pagination, context, include and exclude
const paged = await wp.searchContent('hello', {
  perPage: 5,
  context: 'embed',
  exclude: [10, 20],
  include: [5, 15],
});
```

Each result has the shape `{ id, title, url, type, subtype }`.

### WPAPI chain search

The `search(query)` starter provides the same features through the fluent builder. Pass the query directly to the starter — a separate chained `.search()` call is not needed:

```ts
// Basic search
const results = await wp.search('hello').perPage(5).get();

// Filter by type and single subtype
const pages = await wp.search('about').param('type', 'post').subtype('page').get();

// Filter by multiple subtypes — bracket notation is handled automatically
const mixed = await wp
  .search('test')
  .param('type', 'post')
  .subtype(['post', 'page', 'book'])
  .perPage(10)
  .get();

// Use context, include, and exclude
const withContext = await wp
  .search('hello')
  .context('embed')
  .include([5, 15])
  .exclude([10, 20])
  .get();
```

## Full WPAPI-compatible syntax

### Root chain starters

- `posts()`
- `pages()`
- `media()`
- `categories()`
- `tags()`
- `users()`
- `comments()`
- `settings()`
- `types()`
- `taxonomies()`
- `statuses()`
- `search(query?)`
- `blocks()`

### Chain query methods

- `id(value)`
- `param(name, value)`
- `params(record)`
- `slug(value)`
- `page(value)`
- `perPage(value)`
- `offset(value)`
- `search(value)`
- `status(value)`
- `author(value)`
- `categories(value)`
- `tags(value)`
- `include(value)`
- `exclude(value)`
- `order('asc' | 'desc')`
- `orderby(value)`
- `context(value)`
- `subtype(value)` — single string or array; arrays use bracket notation (`subtype[]=…`)
- `fields(value)` — single field name or array; restricts the response via `_fields`
- `password(value)`
- `embed()`
- `setHeaders(...)` / `headers(...)`
- `toString()`
- `get()`
- `create(payload)`
- `update(payload)`
- `delete({ force? })`
- thenable support: `await wp.posts().slug('hello-world')`

WPAPI chains intentionally block auth-like header overrides via `setHeaders` (for example `Authorization`, `Cookie`, and `X-WP-Nonce`). Use a different `WordPressClient` instance when auth settings need to change.

When `param(name, value)` receives an array it encodes each element using WordPress bracket notation (`name[]=a&name[]=b`). This is the correct format for multi-value parameters on the search endpoint (e.g. `subtype`) and is also accepted by WordPress for other array params such as `categories`, `tags`, `include`, and `exclude`.

### Namespace and route helpers

- `route(resource, namespace?)`
- `namespace('my-plugin/v1').route('sync')`
- `namespace('my-plugin/v1').resource('sync')`
- `registerRoute('my-plugin/v1', '/authors/(?P<id>)')`

## Validation

- Core mutation helpers validate responses with built-in schemas by default.
- You can override defaults by passing any Standard Schema-compatible validator to `responseSchema`, `inputSchema`, or `outputSchema`.

```ts
import * as v from 'valibot';

const postSchema = v.object({
  id: v.number(),
  slug: v.string(),
  status: v.string(),
});

const created = await wp.createPost(
  { title: 'Schema validated', status: 'draft' },
  postSchema,
);
```

- Use `fluent-wp-client/zod` when you need native Zod APIs like `.extend()` or `.parse()`.
- For full patterns and decision guidance, see [Validation guide](./validation.mdx).

## Pagination guidance

- WordPress caps `per_page` at `100`.
- Use `getAll*()` helpers for complete datasets.
- Use `get*Paginated()` helpers when you need totals and page metadata.

## Error handling guidance

- API errors throw `WordPressApiError`.
- Validation errors throw `WordPressSchemaValidationError`.
- Cross-origin absolute URLs are rejected before requests are sent.
